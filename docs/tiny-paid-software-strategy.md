# Tiny, Useful, Paid Software

## Summary

I want to build small, beautiful software products that do one job well.

The goal is not to build a large VC-style SaaS company.

The goal is to build high-quality paid software that can support a solo developer lifestyle business.

Jotter.md is the first product in that direction.

The broader shift is from a services business toward a product and software business.

Services can generate cash, but they mostly sell time.

Products are closer to the work I actually love.

I like building software.

I like product taste.

I like small tools that feel good to use.

## Business Shape

The business should have three parts.

1. YouTube for trust, reach, and distribution.
2. Community, courses, and digital products for the main income.
3. Small paid software products for long-term leverage and product proof.

The current main product is AI Engineer.

AI Engineer is the community and education business.

It teaches developers and technical builders how to build and ship real software with AI using engineering judgment.

The software products should support that larger business.

They should also be real products in their own right.

## Long-Term Goal

The long-term target is roughly $20k to $50k per month.

The first freedom target is around $10k per month.

At that level, I can quit my job and work full time on the work I actually want to do.

The community is currently around $1,500 per month and trending upward.

The main short-term priority is growing the community and education business.

Software is the long-term compounding layer.

## Product Philosophy

The phrase is:

```txt
Tiny, useful, paid software.
```

Each product should be small enough for one person to build and maintain.

Each product should solve one clear problem.

Each product should have strong taste.

Each product should be useful enough that people are happy to pay a small amount for it.

Each product should have low infrastructure cost and low support load.

Each product should make sense even if it only reaches a few hundred paying customers.

The goal is not to make every product huge.

A product that makes $1k to $5k per month can be a win.

For example, $2k per month from a small tool is meaningful.

It is not the whole business, but it is a real asset.

## Why This Is Not Deluded

Building another Markdown app sounds bad if the pitch is just "another Markdown app."

That is not the pitch.

The pitch is a small, excellent tool for a specific workflow.

Many product categories are crowded.

Todo apps are crowded.

Calendar apps are crowded.

Writing apps are crowded.

People still pay for specific products when they like the feel, trust the maker, and use the product often.

TeuxDeux is a useful reference.

It is one of many todo apps.

It still works because it has a clear point of view and people like using it.

Jotter.md should follow that kind of logic.

It should not try to win by having the most features.

It should win by being the nicest small tool for a specific job.

## Jotter.md

Jotter.md is the first product.

The short description is:

```txt
The nicest place to write Markdown that agents can also use.
```

The product is a calm Markdown notepad for humans and agents.

The human side is a beautiful browser writing surface.

The agent side is saved docs, raw `.md` URLs, a CLI, and an API.

The first job is simple:

```txt
Write Markdown beautifully.
Then let agents use it reliably.
```

It should not become a full knowledge base.

It should not become a rich-text editor.

It should not become a team docs platform.

It should not become a project management tool.

## Free And Paid Boundary

Free should be useful immediately.

Free should get people hooked on the writing surface.

Free should not require an account.

Free should be:

- Browser editor.
- Local transient draft.
- Markdown preview.
- Copy Markdown.
- Anonymous fragment share links.
- Local CLI helpers such as opening or previewing a local Markdown file in the browser.

Paid should start when the user wants persistence, portability, agent access, or customization.

Paid should be:

- Saved docs.
- Sync.
- Export.
- Durable public share links.
- Raw `.md` URLs.
- Unbranded public share pages.
- CLI writes to hosted docs.
- API access.
- Agent skill support.
- Dark mode.
- Themes.

The CLI and agent skill can be free to install.

Hosted CLI and API actions should require a paid account.

Free CLI actions should stay local.

For example, `jotter open notes.md` can open a local Markdown file in the browser without uploading it.

For example, `jotter preview notes.md` can open a local Markdown file directly in preview mode.

That makes the tool useful without giving away hosted value.

## Pricing

The product should feel cheap enough that upgrading is obvious once someone likes it.

The current rough target is:

```txt
$5/month
$49/year
```

Another possible later price is:

```txt
$8/month
$72/year
```

The early posture should be low-friction.

The product should feel like an easy individual purchase.

It should not feel like enterprise software.

It should not need procurement or a team budget.

At $5/month:

```txt
400 customers = $2k/month
1,000 customers = $5k/month
```

At $10/month:

```txt
200 customers = $2k/month
500 customers = $5k/month
```

Those are hard numbers, but they are not absurd numbers for a creator-led software product.

## Distribution

YouTube is the main distribution channel.

The goal is to reach 100k subscribers over time.

Products should show up naturally in videos.

The best marketing is building real software in public and showing useful workflows.

Jotter.md can be part of content about:

- Building a small SaaS with agents.
- Designing a focused product.
- Adding auth, Stripe, billing, and entitlements.
- Building a CLI for agents.
- Using raw Markdown URLs as agent context.
- Deploying a simple product on Google Cloud.
- Running a calm, tiny software business.

Ads may be useful later for retargeting or testing positioning.

Ads should not be the main early acquisition channel.

The stronger engine is audience, trust, demos, and useful public work.

## Strategic Role

Jotter.md does not need to become the main business.

It can still be very valuable.

It can become:

- A product I use myself.
- A small revenue asset.
- A SaaS case study.
- A live reference for AI Engineer.
- A source of YouTube content.
- A proof point that I build real software, not just talk about it.

That is the main reason this is interesting.

It connects product, software, content, and education.

It helps move the business away from services.

It creates a path toward a product company that still fits a solo developer.

## Operating Rules

Keep the product small.

Do not add collaboration before the simple sharing model works.

Do not add teams in V1.

Do not add AI chat inside the editor.

Do not add complex rich-text behavior.

Do not make free mode feel broken.

Do not make paid mode feel expensive.

Do not chase enterprise needs.

Build the thing I want to use first.

Let the product become public only after the core workflow feels excellent.

The standard is:

```txt
One job.
Great taste.
Useful every week.
Cheap to run.
Easy to pay for.
Small enough to maintain.
```
